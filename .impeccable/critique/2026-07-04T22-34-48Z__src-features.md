## Anti-Patterns Verdict

**Not AI slop.** A user fluent in Linear/Notion/Stripe would broadly trust this. It has a real, intentional design system — a purpose-built "night" charcoal ramp with documented semantic roles, disciplined loot-gold accent, item-quality "arcane" secondary, named easing tokens, and a complete `prefers-reduced-motion` fallback. Decoration serves the theme; the theme serves a real audience.

**Deterministic scan: CLEAN.** The bundled detector ran over 19 `.tsx`/`.css` files → **0 findings, exit 0** (validated functional via self-test). `global.css` uses impeccable-approved easings (no bounce/overshoot), honors reduced-motion, and uses Mantine CSS variables instead of hardcoded slop. This confirms the "not slop" verdict deterministically.

**Where a discerning user pauses:** the user-visible URL typo `/loot-asigner`; borderline-failing dimmed text contrast; a mouse-only chart with no clickable affordance; a raw `error.name: error.message` string; and one tonal wobble ("No items looted 🥺").

**Visual overlays:** not collected — no dev server was running and starting one was out of scope. No rendered-DOM/screenshot evidence; contrast was computed statically from the theme hex ramps instead.

## Overall Impression

The bones are genuinely good. This clears the slop bar by a wide margin and has real craft in its theming and motion. What holds it back from "trustworthy tool" is **execution gaps at the highest-stakes moments**: writes are irreversible, errors are handled coldly or silently, and the core fairness data (per-player loot history) is locked behind a mouse-only chart. The single biggest opportunity: make the officer's write path *safe* (undo + confirmation) and the reader's data path *accessible*.

## What's Working

1. **A real design system, not defaults** — the semantic `night` ramp, gold primary with `autoContrast`, and arcane accent give a coherent identity. This is why it isn't slop, and the clean detector scan backs it up.
2. **Motion done right** — named easing tokens, press-feedback, hover-lift, staggered entrance, and a full `prefers-reduced-motion` block. Rare craft at this scale.
3. **Thorough state coverage on the read path** — Statistics handles pending / error / empty / populated; Loot Assigner distinguishes no-query vs no-results with different icons and copy; loading uses a blur overlay, not a layout-shifting spinner.

## Priority Issues

**[P0] No undo / reversibility for irreversible writes**
- Why it matters: A loot council uses this fast, post-raid, at odd hours. Mis-assignments are inevitable and permanent from the UI — anxiety at the exact peak moment (assigning a contested drop).
- Fix: Add an "Undo" action to the success toast (Mantine notifications support actions) and/or an unassign control in a per-player view. At minimum, a confirmation step showing item + selected players before commit.
- Suggested command: `/impeccable harden`

**[P1] Dimmed / gray text fails WCAG AA on card surfaces** (both assessments agree)
- Evidence: dimmed `#8a8781` on card `#232220` ≈ **4.44:1** (fails 4.5:1); passes marginally on body bg (~4.86:1). Also flagged by the detector-side contrast pass: arcane-6 `#6926ff` as text ≈ 2.5–2.75:1 (fails), dark-3 `#66635e` ≈ 2.7–2.9:1 (text-fail), `withBorder` card borders ≈ 1.4–1.5:1 (below 3:1 for UI boundaries).
- Why it matters: AA is the stated target and dimmed text is pervasive and small (subtitles, hints, stat labels). Fails Sam and anyone on mobile in bright light.
- Fix: Lighten `night[2]` toward ~`#9a9791`+ (≥4.5:1 on cards) or stop using `dimmed` for essential text; never use arcane-6/dark-3 as foreground text on dark; bump chart ticks to a lighter gray.
- Suggested command: `/impeccable colorize` (or `/impeccable audit` to sweep every instance)

**[P1] Loot-per-player chart is mouse-only and screen-reader-invisible**
- Why it matters: Clicking a Recharts bar is the *only* path to a player's loot history — core fairness data. Bars aren't keyboard-focusable, the SVG has no `role`/`aria-label`, and there's no visible "click a bar" hint or table alternative. Keyboard/SR users (Sam) are locked out; everyone loses the affordance.
- Fix: Add a focusable player row/table that opens the same modal, add `aria-label`s and `role="img"` to the chart, and a visible hint ("Select a player to view their loot").
- Suggested command: `/impeccable harden` (+ `/impeccable clarify` for the affordance hint)

**[P1] Errors mishandled: raw strings on read, total silence on search**
- Why it matters: Statistics renders `{error.name}: {error.message}` verbatim; the item-search mutation on failure falls through to "No items found" — misattributing a network error to a spelling mistake. On flaky mobile (Casey) this is the common path.
- Fix: Human, in-world error copy + a Retry button on Statistics; detect the search-mutation error explicitly and show a distinct error state / toast with retry.
- Suggested command: `/impeccable clarify` (+ `/impeccable harden` for retry paths)

**[P2] User-visible route typo `/loot-asigner`**
- Why it matters: The misspelled route is shareable/bookmarkable and is the clearest "unpolished" tell.
- Fix: Rename to `/loot-assigner` (update `Router.tsx` + `Navigation.tsx`), add a redirect from the old spelling. (Internal typos `PriorirtyTrackerTable`, `playersPriortyEntris` too.)
- Suggested command: `/impeccable polish`

## Persona Red Flags

**Alex (impatient power-user officer)**
- No bulk assign — one search → one modal → one item; recording a full raid's drops is a repetitive grind.
- No undo on mis-assign (P0) — the fast operator is exactly who fat-fingers the MultiSelect.
- Assign confirmation shows only the item *name* — no icon/quality/ilvl to confirm it's the right variant.
- Re-authenticates every token expiry; no keyboard shortcut to jump to search.

**Sam (keyboard + screen reader + contrast)**
- Chart data unreachable by keyboard/SR (P1).
- Dimmed/subtitle text ~4.44:1 on cards fails AA (P1) — nearly every descriptive line.
- Mobile `<Burger>` toggle has no `aria-label` — announced as a bare "button".
- Recharts SVG has no `role="img"`/`aria-label`.
- Positives: modals have focus-trap + Esc + `data-autofocus`; reduced-motion fully honored.

**Casey (distracted mobile)**
- Every nav switch and login needs opening the burger first — no persistent tab/bottom nav.
- Raw error Alert with no Retry on flaky connections; silent search failure reads as "no results".
- Fixed 540px vertical chart grows tall with roster size; `barSize={14}` bars are thin tap targets.

## Minor Observations

- "No items looted 🥺" emoji breaks the WoW register — swap for in-world copy.
- `LootList.tsx` exports `ItemList`/`ItemListProps` (name mismatch).
- Priority table header "Players" vs cell rendering `element.items` → confirm label matches data.
- `key={index}` anti-pattern in the priority items map.
- `Player.lootedCount` duplicated with `lootedItems.length` — redundant source of truth.
- Pagination guard `{page && ...}` is always truthy; a zero-page Pagination can render.
- Top-looter StatCard `wrap="nowrap"` — a long character name could clip against the crown icon.
- Blizzard `CLIENT_SECRET` hardcoded/shipped in `endpoints.ts` — a source-viewing user loses trust.

## Questions to Consider

1. When an officer mis-assigns a contested drop at 1am, where does "undo" live in their mental model — or do they edit the Azure table by hand?
2. Is it acceptable that a *fairness* tool can't audit a player's loot history with a keyboard or screen reader?
3. A single shared access code makes every write anonymous — the app records which *player* looted, not which *officer* assigned. Is that the right identity model for a trust tool?
4. Should read-only guildmates see a prominent "Login" button, or does it invite "why can't I do anything?" confusion?
